# iOS 受信側（見る→わかる→返す）の実現経路マップ

最終更新: 2026-07-04
ステータス: ブレスト整理（実装未着手）。macOS 版 M4「見る→わかる→返す」の iOS 展開の設計メモ。
**優先度: 低（§0 の戦略評価参照。iOS の主戦場は入力側＝キーボード拡張。本マップは
「波が来た時のための仕込み」として保持する）**

---

## 0. 戦略評価（2026-07-04 オーナー判断）— なぜ iOS 受信側の優先度が低いか

1. **汎用チャット AI が「全ての目・全ての耳」になれてしまう**: スマホでは ChatGPT 等に
   スクショを貼れば何でも説明してくれる。「見る→わかる」単体は既にコモディティであり、
   I//O が呼び出しジェスチャを工夫して提供しても差別化にならない。
2. **スマホで「分からないこと」を無理に解決しようとしない**: 認知負荷の高いタスク
   （役所の手続き・保険の契約等）はそもそもスマホでやらない／後回しにする行動が自然。
   受信側が価値になる閾値は「説明してくれる」ではなく**「自律的に入力フォームを
   埋めてくれる」レベル**（= master plan「M4 の先」階段 2〜5 相当）。そこに届かない
   中間実装は使われない。
3. **業務ハーネスの価値がスマホには無い**: PC では業務利用という文脈が I//O の
   ハーネス（摩擦ゼロの capture→interpret→inject ループ）を価値にするが、スマホで
   業務をする人は少なく、ユースケースは個人利用に寄る。
4. **したがって iOS の価値提供点は入力の瞬間**: アプリケーションレイヤーよりさらに
   ユーザーに近い、最も近い接点＝**キーボード拡張**。ここは汎用 AI アプリが構造的に
   占有できないポジションであり（ChatGPT はキーボードになれない）、Persona / Relationship /
   トゲ取りという I//O 固有の資産がそのまま効く。

**結論**: iOS は入力側（キーボード＋音声＋レビュー）に集中する。受信側は
(a) Safari Web Extension が「説明」でなく「フォームを埋める」レベルで成立する目処が
立った時、または (b) computer use 系の波（master plan の私設ベンチ基準）が来た時に、
本マップを取り出して再評価する。

macOS 版は「右Shift2回 → 全画面キャプチャ → VLM 解釈 → 提案アクション → 承認して注入」まで
実機で動いている（[master plan](../../app-mac/docs/universal-io-master-plan.md) §M4）。
このドキュメントは、同じ体験を iOS の制約下でどう成立させるかの選択肢を整理する。

## 1. 前提: iOS の「穴」は実質 4 つ

iOS でサードパーティが他アプリの画面・コンテンツに触れる公式経路は、次の 4 つに集約される。
それ以外は審査外ルート（私的 API）でも**技術的に塞がっている**（サンドボックスは審査ではなく
カーネルで強制されるため。脱獄を除く）。**Broadcast Extension が事実上の技術上限であり、
幸いそれは審査内**である。

1. **スクリーンショット** — ユーザーが撮る → 写真ライブラリ／共有シート経由で受け取る
2. **ReplayKit Broadcast Upload Extension** — システム全体の画面ストリームを受け取れる唯一の API
3. **Safari Web Extension** — Safari 内のページ DOM を読み書きできる
4. **クリップボード／共有シート** — ユーザーが明示的に渡すもの

## 2. 経路の選択肢

### A. スクショ駆動 3 点セット（本命・v1。審査リスク: ほぼゼロ）

macOS の「右Shift2回 → 全画面」に最も近い相似形。

```
呼び出しジェスチャ（§3 参照）
  → ショートカット「スクリーンショットを撮る」（公式アクション）
  → I//O の App Intent に画像を渡す（openAppWhenRun=false でバックグラウンド実行可）
  → Gateway /api/ai/vision へ送信（macOS 版と共通・実装済み）
  → 結果を通知 / Live Activity / アプリで「状況→求められていること→提案アクション」表示
```

- すべて公開 API。ショートカットはユーザーへ 1 タップ配布可能（.shortcut）。
- 補助導線:
  - **Share Extension**: スクショ直後のサムネイル → 共有 → I//O（正攻法）
  - **カメラロール駆動**: I//O を開いた瞬間に最新スクショを自動検出して「これを読みますか？」
    （PHPhotoLibrary。Limited Access で「スクリーンショットのみ許可」も可能）

**「返す」の閉じ方が iOS ならでは**: macOS は ⌘V 合成注入だが、iOS では**キーボード拡張自体が
注入器**。Vision 解釈の返信文案を App Group に置く → ユーザーがどのアプリでも入力欄をタップ →
I//O キーボード上部に文案チップ → タップで注入。既存の音声 IPC 基盤（`SharedStore`）が
そのまま使える。「見る→わかる→返す」の承認駆動ループが iOS でも閉じる。

### B. Safari Web Extension（隠れた本丸・v2。審査リスク: 低）

iOS 15+ の Safari Web Extension はコンテンツスクリプトで**ページ DOM を読める・書ける**。
Web に限れば:

- 「見ているものを読む」= DOM テキスト取得（スクショ→VLM より高精度・低コスト・低単価）
- **「fill_form」= フォームへの直接書き込みが今日成立する**

master plan「M4 の先」階段 2（Vision × AX の fill_form）が、iOS では Safari 内でなら
macOS より先に実現できる。役所の電子申請・病院の Web 問診・各種手続きは大半がブラウザなので、
「認知の義肢」の本丸ユースケースと守備範囲がほぼ一致する。
擬似ブラウザ（WKWebView アプリ）の自作より、ユーザーが普段の Safari のままでいられる
こちらが筋が良い（セッション・パスワード・履歴が本物の Safari に残る）。

### C. Broadcast Upload Extension ＋ PiP（アクロバット枠。審査リスク: 中〜高）

- **見る側**: コントロールセンターの画面収録（長押し → 配信先に I//O を選択）で、
  **全アプリの画面フレームがリアルタイムに Extension へ流れ込む**。Zoom・Discord・
  TeamViewer QuickSupport が使う公認経路。メモリ 50MB 制限のため、フレームは
  画面変化検知＋ ~1fps に間引き・縮小して Gateway へ。
- **見せる側**: AVPictureInPictureController に自前レンダリングした映像を流すと
  **他アプリの上にフローティング表示**できる（翻訳字幕系アプリが実際にやっている転用。
  PiP の建前は動画再生なので審査リスクあり。前例はある）。
- 組み合わせ = 「**セッション型の認知の眼鏡**」: セッション開始後は、どのアプリを見ても
  画面の解釈が PiP に浮かび続ける。iOS で実現可能な形として最も北極星に近い。
  録画中インジケータが常時出るのはプライバシー約束（master plan §3.4）的にはむしろ正しい。

### D. iOS 26 Visual Intelligence 連携（OS 公式の口。要最新ドキュメント確認）

iOS 26 ではスクリーンショット画面に Visual Intelligence が統合され、サードパーティアプリが
App Intents 経由でその UI に登場できる枠がある（2026-07 時点では画像検索プロバイダとしての
統合が中心）。ここに乗れれば A の導線が OS 公式に格上げされる。API の守備範囲は実装前に
最新ドキュメントを確認すること。

### E. その他

- **コピー駆動キーボード**: Full Access キーボードはペーストボードを読める。受信文をコピー →
  入力欄タップ → キーボードに「整理して読む」。macOS の「選択して右Shift2回」に最も近い
  操作感だが、iOS 16+ はペースト確認ダイアログが挟まる摩擦あり。
- **カメラ駆動**: 紙の書類・他デバイスの画面・現実のフォームを DataScanner / Live Text で。
  役所・病院の「紙」はこちらが実態という可能性もある。
- **審査外ルート**: EU 代替マーケット等はあるが、iOS は審査外でも技術的に穴が増えない（§1）。

## 3. 呼び出しジェスチャ（「困った瞬間」からモードに入る手）

iOS には macOS の右Shift2回のような**グローバルホットキーが存在しない**。
LINE / Slack / Gmail を見ていて「分からない」が起きた瞬間に使える 1 アクション候補:

| トリガー | 対応機種 | 手数 | 信頼性 | 備考 |
|---|---|---|---|---|
| **背面タップ（Back Tap）** | iPhone 8+ | トントン（1 アクション） | 中（誤発火・不発あり） | 設定→アクセシビリティ→タッチ→背面タップにショートカット割当。右Shift2回に最も近い体感 |
| **アクションボタン** | iPhone 15 Pro+ | 長押し（1 アクション） | 高 | 物理ボタン。対応機種限定 |
| **通常のスクショ**（電源＋音量上） | 全機種 | 撮る→共有 or アプリへ（2 手） | 高 | 既に体に染みついた動作。共有シート or カメラロール駆動で受ける |
| **コントロールセンター** | iOS 18+ | 引き下ろす→タップ（2 手） | 高 | ControlWidget API でサードパーティのボタンを常設可能。ロック画面ボタンにも割当可 |
| **Siri** | 全機種 | 「Hey Siri, アイオーで読んで」 | 中 | App Intents は Siri から音声起動できる。ハンズフリー・アクセシビリティ的に重要 |

**設計方針（案）**: 主役は「スクショを撮る」という既存習慣に乗ること（信頼性が最も高く
教育コストゼロ）。その上で、背面タップ／アクションボタン／コントロールセンターを
「上級者向けの 1 アクション化」としてオンボーディングで提示する。
C（Broadcast セッション）だけは性質が違い、**困る「前」にセッションを開始しておく**常用型。

## 4. 推奨の組み立て

| フェーズ | 経路 | 一言 |
|---|---|---|
| v1 | **A: スクショ駆動**（トリガー複数＋App Intent＋Share Extension）＋キーボード注入で承認ループを閉じる | Gateway `/api/ai/vision`・解釈スキーマ・Persona/Relationship 注入がそのまま使える。追加サーバー開発ほぼゼロ |
| v2 | **B: Safari Web Extension** | Web の fill_form。「認知の義肢」の実利が最速で出る |
| 実験枠 | **C: Broadcast＋PiP** | 北極星のプロトタイプ。審査は挑戦だが技術は公認 |
| 監視 | **D: Visual Intelligence** | OS が口を開けてくれるのを待って乗る |

**共通資産**: macOS 版で作った Gateway の vision エンドポイント・解釈スキーマ
（situation / extracted / asks / suggested_actions）・Persona / Relationship 注入は
A〜C すべての「わかる→返す」部分で共通に使える。iOS 固有なのは「見る」の取得経路と
「返す」の注入面（＝キーボード拡張）だけで、注入面は既に POC が動いている。
